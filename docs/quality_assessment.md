# 数据质量评估与优化路线

## 1. 当前结论

华夏地名谱当前已经具备可用的基础地名底座：省、市、县、乡镇/街道、村/社区五级层级完整，父子关系无断链，地名代码无重复，别名索引可处理常见村委会/居委会/社区后缀差异。

更准确地说：当前“基础数据底座”已经适合 Agent 做消歧、行政下钻、同名统计和按需补齐地名由来；但“地名由来知识库”还只是起步阶段，目前只有少量样例，需要后续批量/按需采集持续扩展。

## 4. 样例校验

### 4.1 武汉市

- 基础库命中：`湖北省/武汉市`
- 代码：`4201`
- 层级：`city`
- 由来知识：已从中国·国家地名信息库补齐。

### 4.2 南高村

用户输入 `南高村` 时，别名表可以关联到基础库中的：

- 本地标准记录：`山东省/东营市/广饶县/广饶街道/南高村委会`
- 本地代码：`370523001227`
- 来源地名信息库记录：`山东省/东营市/广饶县/南高村`

这说明别名归一策略能解决 `南高村` vs `南高村委会` 这类常见差异。

## 5. 当前覆盖是否全面

### 5.1 基础行政地名覆盖

当前五级基础底表覆盖面较好：

- 覆盖全国 31 个省级行政区。
- 覆盖到乡镇/街道级。
- 覆盖到村/社区级。
- 支持父子层级、同名地名消歧和别名归一。

但需要注意：

- 当前五级底表主要基于 2023 年统计用区划代码数据，后续行政区划变更不会自动反映。
- 村级数据是统计口径，不完全等同于民政/地名库口径。
- 港澳台没有纳入当前五级主表。
- 道路街巷、自然地理实体、历史地名还未成为本地基础底表。

### 5.2 地名由来覆盖

地名由来仍处于持续采集阶段，省级和可采集地级层已完成，县级层正在批量补齐：

- 当前 `place_knowledge` 有 1,546 条，已补齐 31 个省级行政区、333 个可采集地级行政区、1,180 个县级行政区，并保留南高村样例。
- 已验证国家地名信息库能稳定返回由来、含义、历史沿革。
- 已支持按需补齐并和本地基础地名关联。

由来知识库的主要问题不是技术可行性，而是规模化采集、来源合规、证据留存和质量控制。

## 6. 当前主要问题

1. 生成物过大：`topo_scholar.sqlite` 超过 1GB，不能直接作为普通 Git 文件上传。
2. 基础数据版本偏旧：五级村级底表来自 2023 版，需要后续接入更权威/更新的数据源。
3. 地名由来样本太少：目前只验证了采集链路，没有形成规模。
4. 地名库口径差异：国家地名信息库中的“行政村”可能对应本地基础库中的“村委会”，需要继续优化匹配规则。
5. 道路街巷未入库：用户需求包含“每一个街道的名字”，这里要区分“行政街道办事处”和“道路街巷”。
6. 全文/语义检索仍弱：目前只是 LIKE 查询，后续需要 FTS/BM25/向量检索。

## 7. 上传策略

为了让其他用户可以直接 clone 后使用，同时避免 GitHub 大文件限制：

- 上传必要原始 CSV 数据。
- 上传小型知识表 `place_knowledge.csv` 和省/市/县三级采集队列 `collection_queue.csv`。
- 上传构建脚本和自动 bootstrap 逻辑。
- 不上传巨大生成物：`places.csv`、`admin_edges.csv`、`topo_scholar.sqlite`。
- 用户 clone 后第一次运行 CLI 时，会自动构建本地 SQLite。

这样能保证仓库可维护，也能保证数据和使用方式可复现。

## 8. 未来优化路线

### P0：基础底座继续打牢

- 增加当前行政区划核验器，对接国家地名信息库/民政口径。
- 引入版本差异表，记录撤销、合并、更名、代码变化。
- 完善港澳台和特殊区域处理。
- 区分行政街道和道路街巷，建立不同实体类型。

### P1：地名由来规模化采集

- 先批量补齐省、市、县三级。
- 再补齐热门城市、历史文化名城、重点县域。
- 对用户查询过的村/社区自动补齐。
- 建立 `confidence`、`evidence_url`、`evidence_quote`、`raw_hash` 审核链。

### P2：检索能力升级

- SQLite FTS5 起步，支持由来/含义/沿革全文检索。
- 后续接入 BM25 和向量检索。
- 增加“命名类型”标签：姓氏、方位、水文、地形、迁徙、屯垦、历史人物、少数民族语等。

### P3：知识质量与产品化

- 处理同名地名歧义和多来源冲突。
- 增加人工审核队列。
- 输出 Agent 回答模板，强制引用来源。
- 做 MCP Server 安装文档和 Skill 使用说明。
