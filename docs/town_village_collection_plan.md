# 乡镇/街道与村/社区由来采集评估

## 1. 结论

下一阶段如果覆盖当前五级基础库中的全部乡镇/街道与村/社区，需要新增约 **661,924** 条地名由来知识：

| 范围 | 基础库数量 | 当前已有由来 | 待采集 |
|---|---:|---:|---:|
| 乡镇/街道级全部实体 | 41,352 | 0 | 41,352 |
| 其中：行政街道办事处 | 9,112 | 0 | 9,112 |
| 村/社区级全部实体 | 620,573 | 1 | 620,572 |
| 合计（乡镇/街道级 + 村/社区级） | 661,925 | 1 | 661,924 |

这里的“街道”有两种口径：

1. **行政街道**：如“东华门街道”，当前基础库已有 9,112 条，属于乡镇/街道级。
2. **道路街巷**：如“中山路”“人民街”“解放巷”，当前基础库尚未收录，不能和行政街道混为一类，需要另建 `road/street_road` 实体和单独来源体系。

因此下一阶段建议先做“行政乡镇/街道 + 村/社区”，道路街巷作为独立工程排期。

## 2. 当前基础库规模拆解

### 2.1 乡镇/街道级

| 类型/后缀 | 数量 | 说明 |
|---|---:|---|
| 镇 | 21,379 | 优先级高，地方志和政府站点相对容易找到 |
| 街道 | 9,112 | 用户感知强，适合作为下一阶段第一批 |
| 乡 | 7,166 | 可按县域批量采集 |
| 民族乡 | 883 | 少数民族语源和迁徙由来价值较高 |
| 苏木/民族苏木 | 154 | 需注意蒙语语源和行政区划口径 |
| 地区/办事处/管委会/社区等非标准项 | 2,658 | 需要单独复核，避免把功能区当成普通乡级政区 |
| 合计 | 41,352 | 当前无系统性由来覆盖 |

### 2.2 村/社区级

| 类型/后缀 | 数量 | 采集难度 |
|---|---:|---|
| 村委会 | 267,376 | 中等，国家地名信息库和县级地名志可能覆盖 |
| 村民委员会等委员会类 | 189,671 | 中等，需要标准化为自然村/行政村名称后匹配 |
| 直接村名、嘎查等 | 36,812 | 中高，需要结合县乡过滤消歧 |
| 社区/居委会类 | 126,714 | 较高，很多是新建社区，名字由来常见于街道办、社区简介或新闻稿 |
| 合计 | 620,573 | 当前仅保留 `南高村` 1 条样例 |

## 3. 采集优先级

### P0：行政街道试点（9,112 条）

目标：先覆盖“街道办事处”级别，因为用户经常问“某街道名字怎么来的”，且数据量可控。

采集策略：

- 使用当前基础库中的 `level=town` 且名称后缀为“街道”的 9,112 条。
- 优先从中国·国家地名信息库按 `省/市/县 + 名称 + 乡级行政区` 检索。
- 未命中时转向区县政府、街道办官网、区县地名志、区县志。
- 输出到独立试点队列，不直接污染当前已完成的省市县队列。

### P1：全部乡镇/街道级（41,352 条）

目标：覆盖镇、乡、民族乡、苏木、地区、办事处等乡级实体。

采集策略：

- 镇/乡/民族乡/苏木：仍以国家地名信息库 + 地方志为主。
- 非标准项（开发区、管理区、办事处、地区）：走人工复核或地方政府来源。
- 对少数民族地区增加语源字段复核，例如蒙语、维吾尔语、藏语、满语、壮语等。

### P2：重点村/社区样本库（先 10,000-50,000 条）

不建议一开始直接全量 620,572 条。原因：

- 村/社区来源分散，自动命中率和错配风险明显高于县级。
- 同名村极多，必须依赖 `省/市/县/乡镇` 四级过滤。
- 全量知识表可能超过 GitHub 普通仓库适合承载的体积，需要先设计分片和压缩发布。

优先采集：

- 用户查询过的村/社区。
- 历史文化名村、传统村落、红色村庄、少数民族特色村寨。
- 每个县至少若干样本，形成全国覆盖的“种子层”。
- 地名来源特别清晰的村，例如姓氏、方位、水文、地形、迁徙、古迹、产业、少数民族语源。

### P3：村/社区全量扩展（620,572 条）

全量扩展前需要先完成：

- 分省/分市知识分片，例如 `data/knowledge_shards/province=山东省/*.csv`。
- SQLite 本地构建支持按需导入分片。
- 来源许可证和转载限制分层声明。
- 自动采集置信度模型：官方/地方志为 high，政府新闻/媒体为 medium，参考站点为 low 或只进待复核表。

## 4. 来源优先级

| 优先级 | 来源 | 用途 | 置信度建议 |
|---|---|---|---|
| S1 | 中国·国家地名信息库 | 标准地名由来、含义、历史沿革 | high |
| S1 | 地方志/地名志/史志办 | 乡村由来、旧称、迁徙史 | high |
| S1 | 区县/乡镇/街道政府官网 | 乡镇街道简介、历史沿革 | high/medium |
| S2 | 政府新闻、融媒体中心、兵团/地方新闻门户 | 直接说明由来的补充来源 | medium/high，视证据强度 |
| S3 | 传统村落、文旅、非遗、红色资源公开资料 | 村名文化解释和故事性来源 | medium |
| S4 | 商业地图、百科、泛目录站 | 仅作发现线索，不直接作为高置信来源 | low/reference |

禁止或谨慎：

- 不抓取需要登录、付费、授权不明的内部资料。
- 不批量爬取高德/百度/腾讯地图作为道路或 POI 底表。
- OpenStreetMap 可作为道路街巷候选来源，但受 ODbL 约束，需要单独许可证和数据隔离。

## 5. 推荐执行方式

### 5.1 队列设计

不要把 661,924 条待采集项一次性塞入当前 `collection_queue.csv`。建议新增分阶段队列：

- `data/processed/collection_queue_town.csv`
- `data/processed/collection_queue_village_seed.csv`
- `data/processed/collection_queue_village_full.csv`（全量阶段再生成，不建议近期提交）

队列字段沿用当前 `collection_queue.csv`，新增可选字段：

- `collection_phase`: `town_street_pilot` / `town_full` / `village_seed` / `village_full`
- `normalized_query_name`: 去除“村委会/居民委员会”等后缀后的检索名
- `source_hint`: `mca_geonames` / `local_gazetteer` / `local_government` / `media`
- `risk_flag`: `high_dup_name` / `nonstandard_area` / `community_new_name`

### 5.2 自动化流程

1. 从 `places.csv` 生成阶段队列。
2. 对名称做检索标准化：
   - `南高村委会` -> `南高村`
   - `多福巷社区居委会` -> `多福巷社区`
   - `卜尔汉图嘎查` -> `卜尔汉图嘎查`
3. 先用国家地名信息库检索，并强制四级过滤：省、市、县、乡镇。
4. 候选超过 1 条或行政层级不匹配时进入 `needs_review`。
5. 保存 `evidence_url`、`evidence_title`、`evidence_quote`、`raw_hash`、`confidence`。
6. 定期导入 `place_knowledge.csv`，重建 SQLite，跑 `validate_data.py`。

### 5.3 并发与反爬

原则：以缓存、限速、断点续跑为主，不做绕过式采集。

- 官方接口建议 `workers=1-3`，`sleep>=1s`，按省分批跑。
- 所有响应写入 `data/cache/`，缓存不提交 Git。
- 遇到 429、验证码、403 或异常跳转，立即降速或停止该来源。
- 不使用多 IP 绕过访问控制；如果需要大规模数据，应优先寻找开放下载、政府公开数据或联系来源方授权。

## 6. 数据入库与仓库体积评估

当前 `place_knowledge.csv` 约 6.9 MB / 3,229 条，平均每条约 2 KB。若 661,924 条全部写入单个 CSV，粗略可能超过 1 GB，不适合直接进入普通 Git 仓库。

推荐策略：

- 主仓库保留核心知识表、样例和小规模高质量分片。
- 大规模村/社区由来使用分省分片、压缩包或 GitHub Release 发布。
- 本地 SQLite 由脚本按需构建，不提交 `topo_scholar.sqlite`。
- 文档中明确数据来源、许可证和置信度，避免用户误以为所有条目同等权威。

## 7. 下一步落地建议

1. 先生成 `行政街道 9,112 条` 试点队列。
2. 跑 100-500 条小批量，估算国家地名信息库命中率和错配率。
3. 根据命中率决定是否扩展到全部乡镇/街道 41,352 条。
4. 同步建设村/社区种子队列，先做 10,000 条高价值样本。
5. 村/社区全量阶段再设计分片发布，不直接塞入当前主知识表。
