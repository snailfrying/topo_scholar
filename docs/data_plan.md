# 数据建设方案

## 1. 数据建设原则

华夏地名谱的数据分两类：

1. 基础底表：行政层级、标准名称、代码、父子关系。要求稳定、可批量导入、可离线检索。
2. 地名知识：由来、含义、历史沿革、旧称、传说、证据。要求可溯源、可持续补齐，不要求第一天全量。

先把基础底表做扎实，再通过官方地名库和搜索能力持续补齐地名由来。

## 2. 数据源优先级

### P0：官方与准官方基础数据

- 中国·国家地名信息库：标准地名、类型、坐标、地名由来、含义、历史沿革。
- 国家统计局统计用区划代码和城乡划分代码：村/居委会级名称和代码基础底表。
- 民政部/国家地名信息库行政区划信息：当前行政区划核验。

### P1：开源基础数据

- `modood/Administrative-divisions-of-China`：五级行政区划 JSON/CSV/SQLite，适合快速启动。
- `xiangyuecn/AreaCity-JsSpider-StatsGov`：四级行政区划、拼音、坐标、边界，适合行政查询和地图能力。

### P2：扩展地名和空间数据

- OpenStreetMap / Geofabrik China：道路、POI、部分边界。
- 地方政府官网、地方志、地名志、文旅/民政资料：补充由来和沿革。

## 3. 第一阶段数据目录规范

```text
data/
  raw/                 # 原始下载数据，不直接修改
  interim/             # 清洗中间产物
  processed/           # 标准化后数据，可导入数据库
  cache/               # 按需 API 响应缓存
  metadata/            # 数据源、许可证、采集日志、hash
```

## 4. 标准化产物

第一阶段输出以下文件：

- `data/processed/places.csv`
- `data/processed/admin_edges.csv`
- `data/processed/source_catalog.csv`

后续增加：

- `data/processed/place_knowledge.csv`
- `data/processed/place_aliases.csv`
- `data/processed/place_geometries.parquet`

## 5. 清洗规则

- 名称保留原始中文标准名，不自行简化。
- 代码按字符串保存，不转整数，避免前导零和长代码问题。
- 同名地名不合并，必须依赖行政路径、代码或来源 ID 区分。
- `full_name` 使用“省/市/县/乡/村”的路径格式。
- 对来源冲突的名称保留多来源记录，并标注 `source_priority`。

## 6. 地名由来补齐策略

### 6.1 按需补齐

当用户查询某个地名由来时：

1. 查本地 `place_knowledge`。
2. 如果没有，查国家地名信息库。
3. 如果仍没有，触发联网搜索地方政府/地方志资料。
4. 抽取由来、含义、沿革和证据 URL。
5. 写入缓存并标注置信度。

### 6.2 批量补热

优先补齐：

1. 省级、地级、县级行政区。
2. 历史文化名城、热门旅游城市。
3. 用户高频查询的乡镇街道和村。
4. 指定试点区域，如湖北/武汉或山东/广饶。

## 7. 质量指标

- 基础行政区划导入成功率：目标 99%+
- 同名消歧准确率：目标 95%+
- 由来资料来源覆盖率：第一阶段不设硬性全量目标，先覆盖高频查询。
- 每条知识来源记录完整率：目标 100%
