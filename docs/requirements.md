# 华夏地名谱（TopoScholar）需求文档

## 1. 项目定位

华夏地名谱（仓库/服务名：`topo_scholar`，MCP 服务名：`topo-scholar-mcp`）是面向 Agent 的中国地名知识基础设施，目标是把中国省、市、县、乡镇/街道、村/社区、道路街巷等地名整理成可检索、可溯源、可持续补全的知识库，并通过 MCP 工具和 RAG 能力提供给 Agent 使用。

项目第一阶段不追求一次性拿到“全国所有地名由来”的完整闭集，而是先建设稳定的基础地名底表、行政层级关系、同名消歧能力和本地缓存机制；地名由来、含义、历史沿革通过官方地名库、地方志、政府网站和联网搜索持续补齐。

## 2. 核心目标

1. 建设全国基础地名库：覆盖省、市、县、乡镇/街道、村/社区，后续扩展道路街巷、自然地理实体、居民点。
2. 建设地名由来知识库：保存地名由来、含义、历史沿革、旧称、证据来源、置信度。
3. 提供混合检索 RAG：结构化行政区划查询 + 关键词检索 + 向量检索 + 地理空间检索。
4. 提供 MCP 工具：让 Agent 能按名称、行政路径、代码、地名类型查询和解释地名。
5. 保证可溯源：每条知识必须记录来源、采集时间、原始数据哈希或证据 URL。

## 3. 名称规范

- 中文产品名：华夏地名谱
- 英文/技术名：TopoScholar
- Python 包名：`topo_scholar`
- MCP 服务名：`topo-scholar-mcp`
- 数据库名建议：`topo_scholar`
- 默认 Agent 角色名：地名学者

## 4. 用户场景

### 4.1 精确查询
用户问：“武汉为什么叫武汉？”

系统返回：标准名称、行政路径、地名类型、名称由来、历史沿革、来源、置信度。

### 4.2 同名消歧
用户问：“南高村的由来是什么？”

系统先返回多个候选：山东东营广饶县南高村、山东滨州邹平市南高村、河北邢台临城县南高村等；如果用户没有限定地区，则要求 Agent 展示候选或按上下文自动选择。

### 4.3 行政下钻
用户问：“武汉市下面有哪些区和街道？”

系统返回行政树，并允许继续查询任一区/街道/社区。

### 4.4 主题研究
用户问：“湖北有哪些因水得名的县？”

系统使用结构化过滤 + 关键词/向量检索，找出名称含义中与江、河、湖、港、溪、洲等相关的地名，并给出证据。

### 4.5 持续补齐
本地没有某个村名由来时，系统通过联网搜索或官方地名信息库按需补充，写入缓存并标注来源。

## 5. 数据范围

### 5.1 第一阶段必须覆盖

- 省级行政区
- 地级行政区
- 县级行政区
- 乡级行政区：镇、乡、民族乡、街道、苏木等
- 村级单位：行政村、社区、居委会、村委会

### 5.2 第二阶段扩展

- 道路街巷
- 自然地理实体：山、河、湖、岛、湾、沟、岭等
- 居民点与历史地名
- 古地名、旧称、别名、少数民族语地名

## 6. 数据字段设计

### 6.1 `places` 基础地名表

| 字段 | 说明 |
|---|---|
| `id` | 内部 UUID 或稳定哈希 |
| `source_id` | 来源系统 ID |
| `code` | 行政区划代码/统计代码/地名代码 |
| `name` | 标准名称 |
| `full_name` | 完整行政路径名称 |
| `level` | 级别：province/city/county/town/village/road/natural |
| `type` | 类型：地级市、市辖区、街道、行政村、道路等 |
| `parent_id` | 父级 ID |
| `parent_code` | 父级代码 |
| `province` | 省级名称 |
| `city` | 地级名称 |
| `county` | 县级名称 |
| `town` | 乡级名称 |
| `lon` | 经度，可为空 |
| `lat` | 纬度，可为空 |
| `source` | 数据来源 |
| `source_version` | 来源版本 |
| `fetched_at` | 采集时间 |

### 6.2 `place_knowledge` 地名知识表

| 字段 | 说明 |
|---|---|
| `place_id` | 关联 `places.id` |
| `origin` | 地名由来 |
| `meaning` | 地名含义 |
| `history` | 历史沿革 |
| `old_names` | 旧称/别名 |
| `evidence_url` | 证据 URL |
| `evidence_title` | 来源标题 |
| `evidence_quote` | 关键证据摘录，注意版权长度 |
| `confidence` | 置信度：high/medium/low |
| `source_type` | official/local_gazetteer/gov_site/osm/manual |
| `raw_hash` | 原始响应或文件哈希 |
| `updated_at` | 更新时间 |

## 7. MCP 工具需求

### 7.1 `resolve_place_name`

输入：`name`、可选 `province`、`city`、`county`、`type`。

输出：候选地名列表、行政路径、类型、代码、置信度。

### 7.2 `get_place_origin`

输入：`place_id` 或 `name + admin_path`。

输出：名称由来、含义、历史沿革、来源、置信度；本地缺失时触发按需补齐流程。

### 7.3 `get_admin_children`

输入：行政区代码或名称。

输出：下级行政单位列表，支持层级过滤。

### 7.4 `search_toponyms`

输入：自然语言问题、行政范围、地名类型、时间/朝代/关键词过滤。

输出：混合检索结果，含解释和证据。

### 7.5 `compare_same_name`

输入：地名。

输出：全国同名地名分布、类型统计、行政路径列表。

## 8. RAG 需求

- 必须支持精确匹配：代码、名称、完整行政路径。
- 必须支持同名消歧：同名村、同名街道、同名道路不能混答。
- 必须支持 BM25/全文检索：按“洪洞迁来”“因山得名”“明洪武”等关键词检索。
- 必须支持向量检索：按语义查找“与移民传说相关的地名”。
- 后续支持地理检索：附近、流域、行政边界内。
- Agent 回答必须带来源；没有可靠来源时必须说明“暂无权威由来资料”。

## 9. 第一阶段里程碑

### M0：基础项目骨架

- 建立 `docs/`、`data/`、`scripts/`。
- 写清数据源、许可、采集方式、字段规范。

### M1：基础行政区划数据入库

- 导入五级行政区划基础数据。
- 统一字段，生成 `places` 初始表。
- 支持按名称和行政路径查询。

### M2：官方地名详情缓存

- 实现国家地名信息库按需查询、详情拉取、缓存。
- 为省、市、县、部分乡镇街道补齐由来/含义/沿革。

### M3：MCP 查询工具

- 提供 `resolve_place_name`、`get_place_origin`、`get_admin_children`。
- 让 Agent 可直接使用本地数据与缓存。

### M4：混合 RAG

- 引入全文检索和向量检索。
- 支持主题研究类问题。

## 10. 合规与质量要求

- 不把不明授权的批量抓取结果作为公开数据集发布。
- 官方接口必须限速、缓存、记录来源，不做高频暴力抓取。
- 对每条由来资料保留来源 URL 和采集时间。
- 对不确定资料使用低置信度，并在回答中显式说明。
- 地图和道路数据必须记录许可证，OSM 数据遵守 ODbL。

## 11. 推荐技术栈

- Python 3.11+
- FastAPI 或 Starlette
- MCP Python SDK
- PostgreSQL + PostGIS + pgvector
- DuckDB/Polars 用于离线数据清洗
- SQLite 用于轻量 MVP
- httpx + tenacity 用于数据采集
- pytest 用于采集器和查询逻辑测试

## 12. 第一版验收标准

- 本地存在可查询的五级行政区划基础数据。
- 同名地名能返回多个候选，不误合并。
- 查询一个已有详情的地名时能返回由来/含义/沿革和来源。
- 查询缺失地名时能返回“基础信息存在，但由来资料待补齐”。
- 文档清楚说明数据源、字段、限制和下一步计划。
